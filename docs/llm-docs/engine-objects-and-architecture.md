Struct layouts, handle namespaces, and accessor-side rules for KOTOR's object/creature/party model.

> Migrated from the agent memory store on 2026-06-14 (memory-system cleanup).
> Each section below is one former memory note, preserved verbatim. Verify

## cserverexoapp_facade_split
_KOTOR server-app pointer (AppManager+0x8) is an 8-byte facade (vtable@0, internal@4); party_table and friends live inside the internal, not the facade_

`CServerExoApp` is the 8-byte facade pattern, mirroring the documented
`CClientExoApp` / `CClientExoAppInternal` split. Layout: `vtable @0`,
`internal_ptr @+0x4`. All state — `party_table`, module state,
combat-round registry, etc. — lives inside `CServerExoAppInternal`,
NOT inside the public facade at `AppManager+0x8`.

The party-table walk specifically:
- `AppManager (*0x7A39FC) + 0x8` → `CServerExoApp*` (facade)
- `CServerExoApp + 0x4` → `CServerExoAppInternal*`     ← REQUIRED INDIRECTION
- `CServerExoAppInternal + 0x1b770` → `CSWPartyTable` (embedded)

Verified by decompiling `CServerExoApp::GetPartyTable @0x004aee70`:
`MOV EAX, [ECX+4]; ADD EAX, 0x1b770; RET` — only matches the layout
with the internal-ptr indirection.

**Why:** I shipped a chain that skipped the `+0x4` indirection
(2026-05-15-ish). It silently returned zero bytes for everything
because `pt_num_members=0` reads as "no active party" — a plausible
state — so the Party.Cache and GetPartyMembers helpers logged `n=0`
in every session and were never caught. The PartySelection panel
walk (2026-05-24) finally exposed it: `pt_avail_npcs[0..8]` was
returning `1` for every slot regardless of recruited state, because
we were reading from random heap inside the facade vtable instead of
the real table.

**How to apply:** When walking server-side state from
`AppManager+0x8`, always go through `+0x4` first.
`acc::engine::GetServerPartyTable()` is the corrected helper; any new
server-side accessor should follow the same pattern.

**FIXED 2026-06-03:** `GetPartyMembers` in engine_player.cpp now calls
`GetServerPartyTable()` instead of the broken inline chain. TWO bugs
were stacked:
1. Facade chain skipped `+0x4` → `pt_num_members` read 0 → `n=0` every
   session (the original symptom).
2. After fixing (1), `pt_member_ids[]` turned out to hold NPC **roster
   slot indices** (0..8: 2=Carth, 6=Mission), NOT object handles — and
   `pt_num_members` counts active *followers* only (PC is the implicit
   leader, not in the array). The old code wrote slot indices straight
   into outHandles, so every handle comparison still missed. Fix: resolve
   each slot via `CSWPartyTable::GetNPCObject(slot,0,1)` (→ `(slot,1,1)`
   fallback) to the live creature's object handle before returning.
Verified live: walking near Carth/Mission, `party-check` showed
`ids=[0x287,0x28d] match=1` and the passive person-cue suppressed.

Caught while suppressing the passive-narrate person cue on party
members (`IsActivePartyMember` in passive_narrate.cpp). Side-effect:
party_cache / combat_query `IsPartyMember` now resolve followers for
the first time — combat short-form party-hit speech (combat.cpp
FlushPending) becomes live, not dead. NOTE the PC is no longer in
GetPartyMembers output (was never reliably there anyway).

## race_enum_too_coarse
_K1's CSWSCreatureStats.race only has HUMAN=6 and DROID=5; Twi'lek / Cathar / Echani / Mandalorian all bucket as HUMAN. Discriminate species via appearance_type instead._

The engine's `enum RACE` has only two values (Lane's `swkotor.exe.h` ~line 3953):
- DROID = 5
- HUMAN = 6

Every humanoid species — Twi'lek, Cathar, Echani, Mandalorian, even Selkath and Rodian in many cases — code-races as HUMAN at the rules level. The `race` enum is a rules concept (used by spells like Death/Stun, AI groupings), not a species discriminator.

**Use `appearance_type` (ushort) for species:**
- CSWSCreature inline `appearance` at +0xa4c (`kCreatureAppearanceTypeOffset`)
- Indexes appearance.2da; row label distinguishes "Commoner_01_Mal_Asian" (human) from "Alien_Twilek_Female_01" (Twi'lek) etc.
- Vanilla K1 has 509 rows; we bake a 512-bit static "is human" mask in `dialog_speech.cpp` generated by `build/dump-2da/Program.cs` (label-prefix classifier with false-negative bias).

**Why:** First learned while building the "Read human-speaker subtitles" toggle (2026-05-30). Naive `race == HUMAN` would suppress Twi'lek lines too — exactly what the user explicitly didn't want.

**How to apply:** Any feature that needs "is this NPC actually human" — costume-aware narration, voice-tag heuristics, party-member-only logic — must read appearance_type, not race. The race enum is only useful for "is this a droid" (race == 5) or as one cheap signal among several.

## csws_area_handles
_+0x190 in CSWSArea is ulong* (object IDs), not CSWSObject** — must resolve via CGameObjectArray::GetGameObject_

`CSWSArea.game_objects @+0x190` and `game_object_count @+0x194` define a `ulong*` array of 32-bit object handles, NOT a `CSWSObject**` pointer array. Confirmed by `docs/llm-docs/re/swkotor.exe.h`: `ulong *game_objects;`. Investigation Q5 originally read it as a CExoArrayList-style pointer array — that interpretation was wrong and wedged the entire Pillar 4 cycle until lay-off 4.

**How to apply:** Iterating area objects requires handle resolution per slot. The chain is:

```
*kAddrAppManagerPtr (0x7A39FC) → AppManager*
  +0x4 → CClientExoApp*    (already used by engine_player)
  +0x8 → CServerExoApp*    (server side; needed for game-object resolution)
CServerExoApp::GetObjectArray() @0x004AED70 → CGameObjectArray*
CGameObjectArray::GetGameObject(id, &out) @0x004D8230 → bool
```

**`GetGameObject` returns `true` on miss, `false` on hit** — opposite of what the name suggests. Decompiled code: "if found, write game_object and return false; if not found, write NULL and return true." Always check `if (!miss && out) return out;` not `if (ok && out)`.

`patches/Accessibility/engine_area.{h,cpp}` has the working `AreaObjectIterator` — use that as the canonical implementation. Sentinel handles 0 and 0xFFFFFFFF skipped before resolution. SEH-guarded throughout because the chain can fault during area unload.

**Why:** Found during Pillar 4 cycle gate test 2026-05-04. The dereference-as-pointer path produced `scanned=219, all kinds=0` because every `+0x8` read landed on garbage memory. Multiple iterations of "diagnostic, fix, retest" tracked it down via headless Ghidra decompilation of `CSWSObject::GetArea`. See `docs/navsystem-progress.md` Phase 2 lay-off 4 entry for the full bug-discovery chain.

## player_in_area_object_list
_Engine includes the player's own creature in CSWSArea.game_objects; classifies as Creature kind so any "is this an NPC" filter accepts it unless explicitly excluded._

The player's own server creature is one of the entries in `CSWSArea.game_objects` (offset `+0x190`). When iterated via `AreaObjectIterator`, it's indistinguishable from any other Creature unless the consumer compares pointer identity against `acc::engine::GetPlayerServerCreature()`.

**Why:** `filter::ObjectMatches(obj, CycleCategory::Npc)` previously checked only `kind == GameObjectKind::Creature`. Phase 3 lay-off 4 (Trigger 2 foremost-in-front) broke because the player creature sat at distance 0 from the listener and almost always won the front-cone foremost slot when the character faced near-east (atan2(0,0)=0, lands in Front when relBearing - playerYaw ∈ [-45°, +45°)). The cue then panned dead-center with no spatial info — ~28% of T2 fires were corrupted this way before the filter was added. T1 was masked by its threshold-crossing gate (dist=0 never crosses), so the bug only became visible once T2 (which fires every tick on identity-change, regardless of motion) shipped.

**How to apply:** Any new consumer iterating `AreaObjectIterator` for player-facing cues, narration, focus, or guidance must skip the player creature. The single source of truth lives at the top of `filter::ObjectMatches` — every cycle / passive-narrate / spatial-change-detector path inherits the exclusion automatically. If you're writing a consumer that bypasses `ObjectMatches`, replicate the `gameObject == GetPlayerServerCreature()` guard or you'll see the same dist-0 self-fire pattern.

## pc_name_lives_in_client_exoapp
_Player creature's CSWSCreatureStats.first_name + tag are both empty in vanilla saves; the chargen-set name is in CClientExoAppInternal::player_character_name and must be read via the dedicated accessor._

The player creature's `CSWSCreatureStats.first_name` CExoLocString and the
generic `tag` CExoString are both empty in vanilla KOTOR 1 saves. Companion
NPCs (Trask, Carth, …) have their first_name / tag populated normally.

**Why:** Chargen writes the chosen name to
`CClientExoAppInternal::player_character_name @+0x294` (CExoString*), not
to the creature stats. The creature struct stays neutral so the same code
path can spawn / despawn the player creature without touching the saved
identity.

**How to apply:** Any feature that needs to display or speak the player's
chosen name (Tab leader announce, "you control X", chargen confirmation,
saved-game listing) MUST use
`acc::engine::GetPlayerCharacterName(buf, bufSize)` — which wraps
`CClientExoApp::GetPlayerCharacterName @0x5edab0` (returns CExoString* via
__thiscall on CClientExoApp). Do NOT chain through `GetObjectName(playerCreature, ...)` — it returns empty for the PC. The dedicated accessor is documented in `docs/navsystems-investigation.md` §"Reading the player's character name". Confirmed live 2026-05-05 from `diag_engine_select` Tab logs: `id=0x7fffffff name=[]` for the player creature.

## chargen_world_transient_unsafe_probes
_Hammering engine getters (esp. GetPlayerCharacterName) every frame during chargen->world wedges the load intermittently_

Pressing Spielen at the end of chargen kicks off the chargen→world
transition. In that window the player slot is half-initialised — calling
engine accessors against it every frame can wedge the engine's load
state machine. Symptom: engine main loop stays alive (input, Esc, quit
dialog all work) but the module load never starts (no intro movie, no
world). Intermittent — sometimes the same DLL load wedges, sometimes it
doesn't; classic race timing.

Confirmed unsafe in the chargen→world transient:

- `CClientExoApp::GetPlayerCharacterName @0x5edab0` — getter for the PC
  name slot. Returns "temp" (placeholder) during the transient. Calling
  every frame intermittently wedges the load. Fixed 2026-05-19 by
  gating `combat_query::TickLeaderChangeAutoAnnounce` on
  `GetPlayerPosition` (same gate `passive_narrate` uses).

- `CClientExoApp::GetObjectName` (reached via
  `GetObjectDisplayNameByHandle` from `engine_area::GetObjectName` on
  Creature kind) — writes through a stack CExoString and trips the /GS
  canary (__fastfail, uncatchable) on the half-initialised player
  handle. Documented in `engine_player.cpp::GetActiveLeaderName`.

**Why:** Engine state machines and string accessors aren't designed to
be polled mid-construction. The chargen→world handoff has a brief
window where the PC's CClientExoApp slot, server handle, and creature
stats are not yet coherent. Probes that read fine in steady-state can
either trip the /GS canary or wedge the load logic depending on what
they touch.

**How to apply:** Any per-frame Tick that calls into a CClientExoApp
accessor (or any engine getter that touches player state) needs a
player-loaded gate. Default to `acc::engine::GetPlayerPosition(unused)`
— same gate `passive_narrate::Tick` and now
`combat_query::TickLeaderChangeAutoAnnounce` use. When adding new
per-tick probes, ask: "does this fire during chargen→world?" If yes
and it touches player state, gate it.

## creature_hp_lives_on_client_stats
_Live current_hp / max_hit_points are CSWCCreature.lvl_up_stats +0x4c/+0x4e shorts — the same struct the character-sheet HP label renders from_

For any creature (PC + companions), HP reads canonically from the
CLIENT-side stats struct via:

  CSWCCreature  (client leader from GetClientLeader())
    +0x2f8  -> CSWCLevelUpStats* (embeds CSWCCreatureStats at +0)
               +0x4c  short  pregame_current_hp  (LIVE current HP)
               +0x4e  short  max_hit_points      (LIVE max HP)

Note the misleading field name "pregame_current_hp" — it's the *live*
current HP the engine writes on every damage/heal event, NOT a
pregame snapshot. The character-sheet HP label renders this same
short.

**Why:** Several wrong-looking paths exist on the server side that
return either constants or garbage:

- `CSWSObject+0xdc` (current_hp short) DOES match the live value via
  some engine mirror, but Lane labels the adjacent `+0xe0`
  (`hit_points`) confusingly and the older codebase used `+0xe0` —
  that's a BASE HP component, not live HP (PC at full health reads
  40 there while the actual current+max is 48/48).
- `CSWSObject::GetMaxHitPoints @0x004d01a0` literally just returns
  `[ECX+0xe0]` — same base-HP field, despite the Lane signature
  claiming `__thiscall(void) -> undefined2`.
- `CSWSCreature::GetMaxHitPoints @0x004ed310` works ONLY for the PC.
  Its prolog gates on `stats[+0x6c] == 1` (= `CSWSCreatureStats.is_pc`),
  jumps to a non-PC branch for companions that returns pointer-sized
  garbage (321781826 ≈ 0x132E2502).
- `CSWSCreatureStats+0x48` is NOT hit_points — it's inside the
  conversation `CResRef` field (offsets in the SERVER-side stats
  struct differ from the CLIENT-side one). Server fields at +0x48/+0x4e
  read as ASCII resref bytes ("aa", "ia") for companions and zeros
  for the PC.

**How to apply:** Any feature that needs live HP / max HP for the
active leader should walk `GetClientLeader() -> +0x2f8 -> +0x4c/+0x4e`
under SEH. Don't go through the server-side struct or the engine
GetMaxHitPoints accessor. Verified live 2026-05-24: PC 48/48,
Bastila 65/75, Carth 50/66 all match the character sheet.

## player_control_toggle
_How to flip the player creature between player-driven and AI-driven; the toggle behind autowalk/interact-blocker resolution_

The "player-control-mode dominance" hypothesis from `docs/navsystem-progress.md` Phase 2 lay-off 6 is confirmed and the toggle is decoded:

- **`CSWPlayerControl::SetEnabled @ 0x006792e0`** is the named API. Single int param: `1` = player-driven, `0` = AI-driven.
- It writes two fields:
  1. `CSWPlayerControl.enabled` at +0xc (on the heap object pointed to by `CClientExoAppInternal.player_control` at +0x2a0).
  2. Calls `CSWCCreature::SwitchMode(creature, mode) @ 0x0060f090`, a 54-byte function that's literally `field162_0x3a8 = mode`. Valid modes: 0=AI, 1=player, 2=driving.
- **`CSWPlayerControlCamRelative::Control @ 0x00679940`** (the per-tick input → creature handler) gates the *movement-application* block on `(player_control.enabled != 0)`. Camera-rotation block runs regardless. So `enabled=0` skips the per-tick movement clobber but keeps camera updates flowing.

**Why:** when `enabled=1`, the per-tick input loop overwrites the creature's movement vector each frame. So even an action that *is* dispatched and would otherwise drive movement gets stomped before the next render. Toggling to 0 lets the dispatched action *survive* the input loop until it completes.

**Important scope correction (2026-05-07):** the toggle is necessary but NOT sufficient for player movement. The 2026-05-04 claim that it also "gets `AddMoveToPointAction` / `ForceMoveToPoint` working on the player" was a misdiagnosis — those primitives are NPC-only. The player creature isn't on the AI scheduler's iteration list, so direct queue writes are silently dropped regardless of toggle state. Player movement actually requires the client-to-server message bus (`CSWCMessage::SendPlayerToServerInput_*`) which the picker uses internally. See `project_player_creature_ignores_ai_moves.md` for the full architecture.

**How to apply:** Call `SetEnabled(0)` before dispatching a player-action that's expected to take multiple ticks (engine-action picker dispatch, anchor-walk dispatch). Watchdog the action to completion, call `SetEnabled(1)` to restore manual control. Permanent disable would regress `turn_announce` (server-side yaw stops updating because creature doesn't move from input) but `camera_announce` survives (camera block runs unconditionally). The natural-experiment confirmation exists in vanilla — Tab swaps active party member and uses this exact toggle path.

**Two lifecycle variants on our `acc::engine::SetPlayerInputEnabled` wrapper:**
- `armAutoRestore=true` (default) — autowalk shape: 3-second auto-restore timer arms; `TickPlayerInputRestore()` from OnUpdate flips back to enabled. Each new disable call extends the window.
- `armAutoRestore=false` — sustained-disable shape: no timer, caller responsible for matching `SetEnabled(true)`. Used by view mode (Phase 4 lay-off 3, verified 2026-05-06) where the disable lasts until the user toggles off. The 3s timer would silently re-enable mid-session and W/S would resume walking — verified bug in `patch-20260506-113051.log`.

No script side effects, no save-game state, no item-repository touches, no NPC-behavior changes from the engine's perspective. World identity is `CSWSCreatureStats.is_pc` at +0x6c, which `SetEnabled` doesn't touch.

## csws_action_queue
_CSWSObject AI action queue offsets — action_nodes at +0xfc, count at internal+0x8; the real \"AI action finished\" signal_


The per-creature AI action queue (move-to-point, use-object, the composite walk-to-then-act behind HandleMouseClickInWorld) lives on CSWSObject (base of the player creature):

- `CSWSObject.action_nodes` at **+0xfc** — `CExoLinkedList<CSWSObjectActionNode>`. Offset confirmed: next field is `field54_0x100`.
- `CExoLinkedList` = single pointer `internal` (+0x0). `CExoLinkedListInternal` = `{ head(+0x0), tail(+0x4), count(+0x8) }`.
- Pending-action count: `internal = *(void**)(creature + 0xfc); count = internal ? *(int*)(internal + 0x8) : 0`.
- `CSWSObject.field48_0xe8` (+0xe8) is the "actions enabled" gate — ClearAllActions @0x004CCD80 only runs when it == 1.

RE'd by decompiling CSWSObject::ClearAllActions @0x004CCD80 (iterates `action_nodes.list`, removes each `CSWSObjectActionNode`, then resets `path_find_info`). swkotor.exe.h gives the field layout.

CORRECTION (tested 2026-06-06): combat DOES populate this queue — depth observed swinging 0..5 during a fight (move-into-range + attack actions), NOT purely CSWSCombatRound as first assumed. Harmless for the input-restore use because combat never arms our freeze (every combat delta logged freezeActive=0), so the restore tick never reads the queue during combat — they never overlap. Talk also enqueues a walk action (depth 0→2 on a 30m walk-to-talk), confirming the leader DOES auto-walk to a talk target via the picker path.

`count == 0` is the authoritative "AI action finished" signal — the principled replacement for the 3s `armAutoRestore` timer in engine_player.cpp (see [[project_distant_npc_dialogue_stuck]]). Watcher must latch "saw count>0 since dispatch" (or a ~200ms grace) before restoring, to dodge the post-dispatch enqueue race, and keep a long timeout ceiling as a never-drains backstop.

CSWPlayerControl::SetEnabled @0x006792E0: writes `enabled` (+0xc) then GetCreatureByGameObjectID(player_id) → SwitchMode(1=player / 0=AI). Trivial; restore = SetPlayerInputEnabled(true). See [[project_player_control_toggle]].

## effect_icons — the sighted buff/debuff icon row (RE'd 2026-07-17)
_CSWSCreature.effect_icons at +0x8f4/+0x8f8; one CEffectIconObject per EFFECTICON effect; effecticon.2da drives sprite + priority; exact sighted-parity source for spoken effect names_

- The portrait's buff/debuff icon row is NOT derived from the raw `CSWSObject.effects` array directly. Force powers and item shields apply an extra effect of type **EFFECTICON=67** whose integer param 0 is a row id in **effecticon.2da** (61 rows: force powers, stims, grenades, energy shields, droid items, flash-mine stun; columns `iconresref`, `good`, `priority`).
- `CSWSEffectListHandler::OnApplyEffectIcon` @0x004e4790: gates on `AsSWSCreature`, reads `CGameEffect::GetInteger(effect, 0)` (the icon id) and `(subtype & 7) != 0`, dedups against the creature's existing icon list, reads IconResRef/Priority/Good from `effecticon`, and inserts a new `CEffectIconObject` priority-sorted into **`CSWSCreature.effect_icons`** — `CExoArrayList<CEffectIconObject*>` with data ptr at **+0x8f4** and size at **+0x8f8** (raw offsets confirmed in OnRemoveEffectIcon @0x004e30d0, which also cross-checks `creature_stats` at the known +0xa74).
- `CEffectIconObject` (0x20 bytes): **+0x0 ushort icon row id**, +0x2 CResRef icon resref, +0x18 ushort priority.
- Reading +0x8f4/+0x8f8 and mapping row id → name is exact sighted parity (same icons, same display order). Mod-side: `engine_offsets.h` `kCreatureEffectIcons*`, `examine_view::EffectIconName` (names resolved from spells.2da name strrefs across all 5 locale TLKs), consumed by `combat_query::BuildEffectIconSummary`.
- `CGameEffect` full layout (swkotor.exe.h has a real body): id, +0x8 type, +0xa subtype, +0xc duration, +0x10 expire_day, +0x14 expire_time, +0x18 creator_id, **+0x1c spell_id**, +0x30 num_integers, +0x34 int* integer_list, then float/string/object lists. spell_id → `examine_view::ResolveSpellName` works for future per-effect power naming.

## starforge_placeable_state — captive Jedi + droid terminals (RE'd 2026-07-17)
_Star Forge module scripts track "placeable used up" in NWScript local boolean 19 on the placeable; readable as bit 19 of the CSWVarTable word at CSWSObject+0x110_
